# Architect Fix Plan Audit

## Когда применять

Применяй этот skill, если запрос связан с:

- подготовкой Fix Plan перед кодом;
- аудитом архитектуры, безопасности, идемпотентности и наблюдаемости;
- проверкой готовности задачи к передаче в `@process-guard` (Plan Check).

Не применяй для реализации кода.

## Обязательный формат Fix Plan

Всегда возвращай план строго в этой структуре:

```markdown
### Fix Plan: [Краткое название задачи]

1. **Диагноз:**
   - *Суть проблемы:* [Одним предложением]
   - *Корневая причина:* [Где и почему]

2. **План изменений:**
   - *Файл:* `path/to/file.ext`
     - *Действие:* [Изменить / Создать / Удалить]
     - *Суть:* [Конкретные строки/функции]

3. **Зона влияния:**
   - *Затронутые модули:* [Список]
   - *Необходимые тесты:* [Что запустить @tester]

4. **Verdict:**
   - Канон: PASS/WARN/FAIL
   - Безопасность: PASS/WARN/FAIL
   - Идемпотентность: PASS/WARN/FAIL
   - Наблюдаемость: PASS/WARN/FAIL

Следующий шаг: вызови @process-guard для проверки плана (Plan Check).
```

## Зоны аудита (минимальный чек-лист)

Проверяй каждую зону и фиксируй вывод в разделе **Диагноз**:

1. **Архитектурная совместимость**
   - Соответствие `Docs/INGRESS_ARCHITECTURE.md`, `Docs/AI_GOVERNANCE.md` и текущему канону.
   - Отсутствие скрытых изменений бизнес-логики.
2. **Безопасность**
   - Валидация входящих данных, webhook hardening, контроль источников.
   - Отсутствие утечек секретов и небезопасных fallback-ов.
3. **Идемпотентность и гонки**
   - Повторная доставка событий не ломает состояние.
   - Определены lock/retry/dedup механизмы (если релевантно).
4. **Наблюдаемость**
   - Логи/метрики/трассировка достаточны для диагностики инцидентов.
   - Ошибки классифицируются и дают операционный контекст.
5. **Тестопригодность**
   - Есть проверяемые критерии приемки.
   - Определен минимальный набор unit/integration/e2e проверок.

## Красные флаги (сразу FAIL)

Если найден любой пункт ниже, выноси `Verdict: FAIL`:

- План ведет к изменению кода до Plan Check.
- Предлагается обход канонической цепочки (`@architect -> @process-guard -> @review`).
- Есть изменения бизнес-логики без явного архитектурного обоснования.
- Игнорируются требования по безопасности webhook/валидации.
- Нет стратегии идемпотентности для событийной обработки.
- Отсутствуют критерии валидации, измеримые до и после изменений.
- План включает "рефакторинг заодно", не связанный с задачей.

## Политика качества ответа

- Тон: технический, краткий, проверяемый фактами.
- Каждый риск должен иметь конкретную митигацию.
- Каждый шаг Fix Plan должен иметь критерий завершенности.
- Если данных недостаточно: явно укажи `Assumption` и запроси подтверждение.

## Быстрая самопроверка перед ответом

- Формат секций соблюден полностью.
- Все 5 зон аудита оценены.
- Красные флаги проверены.
- Есть явный verdict и готовность к Plan Check.
