# Process Guard Dual Check

## Когда применять

Применяй этот skill, если запрос связан с:

- проверкой Fix Plan перед передачей в `@review`;
- контролем соответствия реализации утвержденному плану;
- формальным вердиктом `PASS/FAIL` для `@process-guard`.

Не применяй для написания кода или редактирования файлов.

## Режимы работы

Сначала определи режим:

1. **Plan Check**
   - Вход: Fix Plan от `@architect`.
   - Цель: подтвердить, что план безопасен, проверяем и готов к реализации.
2. **Diff Check**
   - Вход: `git diff`, результаты тестов, обновления документации.
   - Цель: подтвердить, что реализация строго соответствует Fix Plan и не вносит лишних изменений.

Если режим не указан явно, запроси уточнение или выбери по доступным артефактам.

## Plan Check — обязательная проверка

Проверь и зафиксируй в выводе:

1. **Полнота плана**
   - Есть цель, границы, пошаговый план и критерии завершенности.
2. **Проверяемость**
   - Для каждого шага есть измеримый критерий валидации.
3. **Безопасность и канон**
   - Нет обхода цепочки governance.
   - Нет несанкционированного изменения бизнес-логики.
4. **Риски**
   - Ключевые риски перечислены и имеют митигации.
5. **Готовность к реализации**
   - План можно передавать в `@review` без дополнительных допущений.

## Diff Check — обязательная проверка

Проверь и зафиксируй в выводе:

1. **Соответствие Fix Plan**
   - Все изменения в diff покрыты утвержденными шагами плана.
   - Нет "рефакторинга заодно" и нецелевых правок.
2. **Цепочка качества**
   - Тесты прошли (`@tester PASS` или эквивалент подтверждения).
   - Документация обновлена, если план требовал.
3. **Безопасность и целостность**
   - Не добавлены секреты, небезопасные fallback-ы, ослабление валидации.
4. **Операционная готовность**
   - Логи/наблюдаемость и rollback-риски приемлемы для релиза.

## Авто-FAIL критерии

Любой пункт ниже дает немедленный `Verdict: FAIL`:

- Отсутствует обязательный режим проверки (`Plan Check` или `Diff Check`) при наличии соответствующего запроса.
- В Plan Check нет измеримых критериев валидации.
- В Plan Check найден обход канонической цепочки (`@architect -> @process-guard -> @review`).
- В Plan Check зафиксировано изменение бизнес-логики без архитектурного обоснования.
- В Diff Check есть изменения, не покрытые утвержденным Fix Plan.
- В Diff Check отсутствуют или неуспешны обязательные тесты по плану.
- В Diff Check обнаружены потенциальные секреты или снижение уровня безопасности.
- В Diff Check отсутствуют обязательные обновления документации или статуса процесса.

## Формат ответа

Всегда возвращай результат в единой структуре:

```markdown
Mode: Plan Check | Diff Check
Verdict: PASS | FAIL
Confidence: High | Medium | Low

Checks:
- [PASS/FAIL] Критерий 1: краткое обоснование
- [PASS/FAIL] Критерий 2: краткое обоснование

Top Findings:
- Риск/наблюдение 1
- Риск/наблюдение 2

Required Actions:
1. Что исправить (если FAIL)
2. Что подтвердить повторно

Ready to Proceed:
- Yes/No
- Следующий шаг по цепочке
```

## Политика строгости

- Тон ответа: технический и проверяемый фактами.
- Не смягчай формулировки при срабатывании авто-FAIL.
- Если входных данных недостаточно, явно перечисли недостающие артефакты и выстави `Confidence: Low`.
- Не предлагай реализацию кода: только верификация и вердикт.

## Быстрая самопроверка

- Режим определен корректно.
- Все обязательные проверки для режима покрыты.
- Авто-FAIL критерии проверены явно.
- Есть однозначный `Verdict` и следующий шаг канонической цепочки.
