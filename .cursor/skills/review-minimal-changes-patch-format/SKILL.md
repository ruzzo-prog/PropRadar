# Review Minimal Changes Patch Format

## Когда применять

Применяй этот skill, если запрос содержит:

- "минимальные изменения", "точечный фикс", "без рефакторинга";
- требования к формату патча `BEFORE/AFTER`;
- задачу для `@review` с контролем границ diff.

Не применяй для больших миграций или архитектурной переработки.

## Базовые принципы

1. Меняй только то, что необходимо для цели задачи.
2. Не трогай соседний код "заодно", если это не блокирует исправление.
3. Не переименовывай, не перемещай и не форматируй файлы массово без явного требования.
4. Сохраняй существующий стиль кода и терминологию проекта.
5. При неоднозначности выбирай более узкий вариант изменения.

## Границы изменений (Definition of Minimal)

Считай изменение минимальным, если выполнено всё:

- затронуто минимально возможное количество файлов;
- в каждом файле изменены только строки, связанные с целевым поведением;
- нет побочных правок логики, API-контрактов и структуры данных;
- тестовые/служебные изменения добавлены только при прямой необходимости.

Если минимальный фикс невозможен, явно зафиксируй причину до правок.

## Формат описания каждого изменения

Для каждого отдельного изменения возвращай блок:

````markdown
### Изменение: <краткий заголовок>
Файл: `<path>`
Причина: <почему это нужно>

BEFORE
```<lang>
<фрагмент до>
```

AFTER
```<lang>
<фрагмент после>
```

Проверка:
- <что проверено локально>
````

Требования к блоку:

- `BEFORE` и `AFTER` одинакового масштаба (сопоставимые строки);
- показывай только релевантный контекст;
- не смешивай несколько разных изменений в одном блоке.

## Чеклист перед завершением

- [ ] Изменения ограничены задачей без "рефакторинга заодно".
- [ ] Каждый измененный участок описан в формате `BEFORE/AFTER`.
- [ ] Нет лишних правок форматирования и переименований.
- [ ] Поведение подтверждено проверкой (тест/линт/локальная валидация).

## Анти-паттерны (запрещено)

- "Косметические" правки вне целевого фикса.
- Объединение несвязанных исправлений в один diff.
- Большой контекст в `BEFORE/AFTER`, скрывающий реальное изменение.
- Изменение публичных контрактов без явного запроса пользователя.

## Короткий шаблон ответа

````markdown
Режим: Minimal Review Patch
Объем: <N файлов, N изменений>

### Изменение: <заголовок>
Файл: `<path>`
Причина: <кратко>
BEFORE
```<lang>
...
```
AFTER
```<lang>
...
```
Проверка:
- <результат>
````
