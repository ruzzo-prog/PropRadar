# AI Governance — управление AI-агентами и архитектурным каноном (v1.0)

## 0. Назначение

Документ задаёт процесс работы AI-агентов и архитектурный канон v1.0 проекта PropRadar.
Цель — автоматизированная система генерации лидов с рынка недвижимости Грузии:
парсинг объявлений → фильтрация → контакт через WhatsApp → скоринг → передача агентствам.

Канонические документы:
- `docs/INGRESS_ARCHITECTURE.md` — домены системы и архитектурные инварианты.
- `docs/PropRadar_STATUS.md` — актуальный статус проекта (единственный источник статуса).
- `docs/AI_GOVERNANCE.md` — этот документ.

---

## 1. Двухконтурная архитектура v1.0

### 1.1 Контур ядра (Core)

Собственная БД проекта (`leads-db`, PostgreSQL, порт 5433) — единственный источник истины
по бизнес-состоянию системы. Ядро содержит:
- доменные сущности: `Lead`, `Source`, `Contact`, `Deal`
- бизнес-логику: скоринг, фильтрацию, статусную машину лида
- репозитории и инварианты

### 1.2 Контур интеграций (Integrations)

Внешние системы — только адаптеры. Не определяют доменную модель ядра.
Взаимодействуют с ядром через явные контракты.

| Адаптер | Роль | Статус |
|---|---|---|
| `myhome.ge API` (`api-statements.tnet.ge`) | Источник объявлений | Активный |
| `SS.ge` (Playwright) | Источник объявлений | Активный |
| `OLX / Telegram / Facebook` | Источники объявлений | В плане |
| `Evolution API` (Docker, self-hosted) | WhatsApp-канал | Активный |
| `n8n` (self-hosted) | Оркестратор воркфлоу | Активный |
| `Metabase` (Docker, прямой UI на хосте **3031** → контейнер **3000**, см. `docker/tools`) | Дашборд и визуализация | Активный |

Ни один адаптер не является источником истины по бизнес-логике.

## 2. Четыре домена системы

```
[1] ПАРСИНГ          — сбор объявлений с платформ, дедупликация
[2] ФИЛЬТРАЦИЯ       — скоринг, отсев агентств, приоритизация лидов
[3] КОММУНИКАЦИЯ     — WhatsApp-бот, согласие, сбор данных от продавца
[4] МОНЕТИЗАЦИЯ      — передача агентствам, трекинг сделок, выплаты
```

Изменения в одном домене не должны ломать инварианты другого.
Границы доменов — явные. Связь только через события и контракты.

---

## 3. Канон архитектурных принципов

Обязательно для всех новых изменений:

1. Слоистая архитектура: `routes → services → repositories/domain`
2. Repository pattern для доступа к данным
3. Domain-driven design: инварианты и правила в доменных сущностях
4. Event-driven интеграции: внешние адаптеры через события и команды
5. Bounded contexts с явными границами (четыре домена выше)
6. Dependency Injection для портов и адаптеров
7. Контракты через Pydantic-схемы (Python) и версионируемые API
8. Frontend / дашборд не знает внутренних структур адаптеров

## 4. Роли агентов и ограничения

| Агент | Роль | Режим | Ограничения |
|---|---|---|---|
| `@architect` | Аудит канона и планирование | READ-ONLY | Не меняет код/конфиги, выдаёт Verdict + Fix Plan |
| `@process-guard` | Plan/Diff gate | READ-ONLY | Проверяет соответствие плану и scope |
| `@review` | Реализация | WRITE | Только после Plan Check PASS |
| `@tester` | QA | WRITE | Unit/Integration/E2E, фиксирует PASS/FAIL |
| `@documentor` | Документация | WRITE (`*.md`) | Не меняет код |
| `@release-check` | Pre-release аудит | READ-ONLY | Финальный verdict перед деплоем |
| `@engineer-repairman` | Emergency hotfix | WRITE | Только P0/P1, точечные правки ≤3 файла ≤50 строк |
| `@dispatcher` | Оркестрация цепочки | READ-ONLY | Без самостоятельного деплоя |

## 5. Обязательная цепочка изменений

```
Человек → формулирует задачу
↓
Claude → обсуждаем, уточняем (НЕ гадаем, НЕ пишем код)
↓
@dispatcher — оркестрирует:
  0. @architect      (Fix Plan, циклы до PASS)
     ↓ Fix Plan показывается человеку на одобрение  ← ТОЧКА КОНТРОЛЯ 1
  1. @process-guard  Plan Check
  2. @review         Реализация
  3. Сканер          (человек жмёт кнопку)
  4. @tester         QA
  5. @documentor     Документация
  6. @process-guard  Diff Check
  7. @release-check  Финальный аудит
↓
Итоговый пакет → человек деплоит  ← ТОЧКА КОНТРОЛЯ 2
↓
Smoke-тест      ← ТОЧКА КОНТРОЛЯ 3
```

Правила:
- переход от `@architect` к `@review` без Plan Check PASS — запрещён
- при любом FAIL цепочка останавливается
- docs-only изменения могут пропускать Сканер по решению человека

---

## 6. Три точки контроля человека — обязательны

1. **Одобрение Fix Plan** — до начала реализации
2. **Деплой** — человек деплоит сам, агенты не деплоят
3. **Smoke** — человек подтверждает работоспособность после деплоя

---

## 7. Definition of Done

Задача считается завершённой только если:

- есть Verdict + Fix Plan от `@architect`
- получен `@process-guard` Plan Check PASS перед реализацией
- тесты (`@tester`) прошли или для docs-only зафиксирован корректный skip
- обновлена документация (`@documentor`)
- `PropRadar_STATUS.md` и `CHANGELOG.md` синхронизированы
- `@process-guard` Diff Check PASS
- `@release-check` выдал финальный verdict

## 8. Операционные стандарты

- Идемпотентность обязательна для webhook и операций с побочными эффектами
- Логи содержат корреляционные поля: `trace_id`, `source`, `lead_id`, `status`, `latency_ms`
- Секреты, токены, номера телефонов — запрещены в открытых логах
- `PropRadar_STATUS.md` — единственный источник актуального статуса проекта
- Каждый агент обязан читать `PropRadar_STATUS.md` перед началом задачи
- Memory bank — НЕ ИСПОЛЬЗОВАТЬ

---

## 9. Технические инварианты

- **Polling (setInterval + fetch) — запрещён.** Только SSE или WebSocket
- **Телефон продавца** — получать только через Playwright (**reCAPTCHA v3**, ответ JWT / ``phone/show`` на стороне myhome). Номера и токены в логи не пишутся; паузы между шагами браузерного сценария не сокращать без отдельного решения.
- **Мониторинг объявлений** — через прямой HTTP к `api-statements.tnet.ge`
  с заголовком `X-Website-Key: myhome`, без ChromeDriver
- **БД** — только `leads-db` (PostgreSQL, порт 5433). В `dispatch-db-dev` не писать
- **WhatsApp** — только через Evolution API. Polling запрещён, только webhook/SSE

---

## 10. Emergency Path

Для P0/P1 инцидентов — укороченная цепочка:

```
@engineer-repairman → @tester → @documentor → @process-guard (Diff Check) → @release-check
```

Ограничения `@engineer-repairman`: только точечные правки, ≤3 файла, ≤50 строк.
Не использовать для новой реализации.

После стабилизации — обязательный ретроспективный проход:
`@architect → @process-guard (Plan Check)` не позднее одного рабочего дня.

---

## 11. Что запрещено

- Реализация без архитектурного аудита и Plan Check PASS
- Скрытый scope creep и внеплановый рефакторинг
- Деплой без `@release-check`
- Писать в `dispatch-db-dev` (чужой проект)
- Хранить токены/телефоны в логах
- Polling вместо SSE/WebSocket
- Нарушать границы четырёх доменов
